## MODIFIED Requirements

### Requirement: 反馈格式和展示
系统SHALL以清晰、易读的格式展示反馈内容，并确保所有类别的反馈数据都能正确解析和显示。

#### Scenario: 结构化反馈展示
- **WHEN** 用户查看单题反馈
- **THEN** 按固定结构展示：语言质量、内容深度、具体建议、参考答案
- **AND** 使用评分、标签、重点标注等方式增强可读性
- **AND** 系统必须正确处理 JSON 格式的反馈数据（无论是字符串还是对象）
- **AND** 当反馈数据格式异常时，显示友好的错误提示而不是空白页面

#### Scenario: 所有类别的反馈正确显示
- **WHEN** 用户查看任何类别（包括 logic-thinking 和 current-affairs）的反馈
- **THEN** 反馈内容必须完整显示，包括评分、优点、待改进、建议、参考思路和参考答案
- **AND** 系统必须统一处理类别名称（logic-thinking 和 logical-thinking 视为同一类别）
- **AND** 后端返回的反馈数据必须是已解析的 JSON 对象，而不是字符串
- **AND** 前端必须添加防御性解析逻辑，处理可能的格式异常情况

#### Scenario: 反馈数据解析容错
- **WHEN** 反馈数据在传输或存储过程中格式异常
- **THEN** 系统必须尝试解析 JSON 字符串（如果接收到的是字符串）
- **AND** 如果解析失败，记录错误日志但不导致页面崩溃
- **AND** 显示友好的错误提示，允许用户重新生成反馈

#### Scenario: 会话总结报告
- **WHEN** 用户查看完整会话的总结
- **THEN** 展示：整体得分、各维度得分、弱点列表、进步对比、下次建议
- **AND** 支持导出为PDF或文本格式
- **AND** 确保所有反馈数据在总结中正确显示
